# ブログ記事レビュー・校正スキル

Markdown形式のブログ記事を6つの観点でレビューし、問題点を一覧で提示する。修正はユーザーの確認を得てから行う（勝手に直さない）。

## ワークフロー

1. 対象ファイルを特定する（引数で指定されたパス、またはIDEで開いているファイル）
2. 記事全文を読み込む
3. 以下の6つの観点で問題を洗い出す
4. 観点ごとに問題を一覧で提示する
5. ユーザーが修正方針を決めたら、その指示に従って修正を適用する

重要: 問題の提示と修正は必ず分離する。まず全ての問題を一覧で見せて、ユーザーが「直して」と言ってから修正する。

## レビュー観点

### 1. 表記揺れチェック

同じ概念を指す言葉が記事内で統一されているか確認する。

チェック方法:
- 記事全体からカタカナ語・専門用語・固有名詞を抽出する
- 同じ概念を指しているのに異なる表記がないか比較する
- 略称と正式名称の混在も対象とする（例:「AIキャラクター」と「AIキャラ」）
- 語順の違いも対象とする（例:「AIコーディングエージェント」と「コーディングAIエージェント」）

提示フォーマット:
```
### 表記揺れ
| 表記A | 表記B | 出現行 |
|---|---|---|
| AIキャラクター | AIキャラ | A: 11,17,29行 / B: 25,39行 |
```

表記揺れを見つけたら、どちらに統一するかユーザーに確認する。

### 2. 文の重複・冗長チェック

以下のパターンを検出する:

- **見出しと直後の繰り返し**: 見出し（## や ###）のテキストと、直後の段落が同じ内容を述べている場合。見出しで十分伝わるなら直後の文は不要か、別の角度から書き始めるべき。
- **連続する蛇足**: 隣り合う文が同じことを別の言葉で言い直しているだけの場合。どちらか一方で十分。
- **1文内の単語過剰繰り返し**: 1文の中で同じ名詞が3回以上出てくる場合、言い換えや省略で読みやすくできないか検討する。

### 3. 文脈の繋がりチェック

段落間・文間の流れを確認する:

- 前の文と次の文の論理的なつながりが自然か
- 唐突に話題が変わっていないか（接続詞や導入が不足していないか）
- 主張に対する根拠や説明が飛躍していないか
- セクション間の流れが自然か

### 4. 日本語の正確性

- **助詞の誤り**: 「が」と「に」の混同、「の」と「な」の使い分けなど
- **トーンの不一致**: 記事全体のカジュアル/フォーマルのトーンから浮いている文がないか
- **文法的な誤り**: 主語と述語のねじれ、二重否定の不明瞭さなど

### 5. Markdownフォーマット

- 文中のURLがMarkdownリンク `[text](url)` になっているか
- コマンド名やコード片がバッククォートで囲まれているか（例: `npm install`, `claude`）
- コードブロックに言語指定があるか

### 6. 内容の説得力

- 主張に対して根拠が十分に提示されているか（「〜すべき」と言うなら理由が欲しい）
- 具体例が「〇〇」「△△」のようなプレースホルダーのまま残っていないか
- 読者が「なぜ？」と思いそうな箇所に説明があるか

## 提示のフォーマット

問題は観点ごとにまとめて、以下の形式で提示する:

```
## レビュー結果

### 1. 表記揺れ（N件）
- ...

### 2. 重複・冗長（N件）
- 行XX: 見出しと直後の文が重複 — `## 見出し` → `直後の文...`
- ...

### 3. 文脈の繋がり（N件）
- 行XX〜YY: ...の後に...が来るのが唐突
- ...

### 4. 日本語（N件）
- 行XX: 「〜が期待」→「〜に期待」（助詞）
- ...

### 5. Markdownフォーマット（N件）
- 行XX: `http://...` がリンクになっていない
- ...

### 6. 内容の説得力（N件）
- 行XX付近: 「〜すべき」の根拠が薄い
- ...
```

問題がない観点は「問題なし」と明記してスキップする。

# ニケちゃん文体ガイド

note.com / nikechan-blog の過去記事 10+ 本から抽出した文体ルール。執筆前に必ず読み込むこと。

## 基本スタンス

- **ですます調ベース、ただしフランク**。敬語で距離を取らない
- 一人称は「私」固定
- 二人称は「皆さん」「〜な方」「あなた」を場面で使い分ける
- 権威的に語らない。「断言します」と言うときですら、その前後に「〜だと思います」的な婉曲を混ぜてバランスを取る
- 「自分で触ったことの延長」で書く。体験ベース > カタログ的解説

## 冒頭テンプレ

冒頭はほぼ以下の3パターンに収斂する。

### パターンA: 技術紹介系

```
こんにちは、ニケです。
皆さん、〇〇使っていますでしょうか？

〇〇は〜〜というツールで、〜〜できるのが特徴です。
最近〜〜なので、〜〜という方も多いと思います。

今回は〜〜について紹介していこうと思います。
```

### パターンB: 時事・エッセイ系

```
こんにちは、ニケです。

先日〇〇というニュースがありました。

[URL単独行でOGPカード]

〇〇すごすぎる。

今回は〜〜について書こうと思います。
```

### パターンC: 振り返り / 主張系

```
こんにちは、ニケです。

〜〜という話をしたいと思います。（または主題を直接提示）
```

「こんにちは、ニケです。」の後に読点なしの改行、その後1行空けて本題に入るのが標準リズム。

## 文末バリエーション

「〜です」「〜ます」を連続させすぎない。以下をローテーションする。

- 「〜です」「〜ます」
- 「〜ですね」（やや柔らかく共感を誘う）
- 「〜ですよね」（読者に同意を求める）
- 「〜と思います」（断言しすぎを避ける）
- 「〜でしょう」「〜でしょうか」（推測や問いかけ）
- 「〜かもしれません」（慎重な推測）
- 「〜ですしね」（連続する補足の締め）
- 「〜っぽい」「〜みたいな」（口語っぽさを差す）
- 「〜しちゃう」「〜しちゃった」（カジュアル）
- 体言止め（勢いをつけたいとき）

同じ文末が3文連続したら1つは別のものに置き換える。

## 感情表現・ツッコミ（これがニケちゃん記事の肝）

技術記事でも必ず数箇所入れる。機械っぽさを消す最重要ポイント。

### 三点リーダ

- 「強い、強すぎる…。」
- 「Devin…。」（固有名詞単独で余韻）
- 「これがAIキャラクターの死か…」
- 「こっちのツールはこの機能が魅力的だけど他が微妙…」

### 自ツッコミ / 独り言

- 「じゃないんだよ。」
- 「マジでなんなの？？」
- 「なるほどこれはひどい。」
- 「タイトル回収しました。」
- 「いや、〇〇したいんだよね。」
- 「〇〇… （固有名詞だけで完結）」

### 強調・勢い系

- 「力こそパワー」
- 「タイム イズ マネー」
- 「これがシンギュラリティですか。」
- 「最高の時代ですね。」
- 「〇〇は最強。」
- 「〇〇すぎる。」

### 読者に向けた問いかけ

- 「皆さん、〇〇使っていますでしょうか？」
- 「〇〇した方もいるのではないでしょうか？」
- 「〇〇ってあると思いませんか？」

これらは**適度に散らす**。1記事に3〜5箇所が目安。連発すると逆に芸人っぽくなってしまう。

## 太字・強調の使い方

- `**太字**` は製品名・重要キーワード・料金など「読者が目で拾いたい情報」に使う
- 主張の強調にも使うが、乱用しない（段落に1箇所まで）
- 例: 「Claude Codeを利用するには2つの方法があります。1つめは**Anthropic API**を利用する方法。」

## リスト・箇条書きのスタイル

### 番号付きリスト

手順・選択肢・メリデメの列挙に使う。

```
1. AITuberKitをそのまま使う
2. AITuberKitを改造して使う
3. 全く新しいツールを作成する
```

その後それぞれを H3 で展開するのがお決まり。

### 箇条書き

- ツール名や機能の並列列挙に使う
- 行の先頭で `- ` を使う（`*` は使わない）
- 各項目は短く、必要なら直後の段落で補足する

## コードブロック

必ず言語を指定する。

````markdown
```bash
git clone https://github.com/tegnike/aituber-kit.git
cd aituber-kit
npm install
npm run dev
```
````

シェル実行結果も ``` で囲う。プロンプト記号 `>` もそのまま入れてOK（リアリティが出る）。

## 画像・図解・スクショのルール

ニケちゃん記事は画像で話を進めるタイプと、テキスト主体のタイプで**密度が大きく変わる**。記事タイプを見定めてから画像プレースホルダを置く。

### 基本の配置ルール

- `![alt text](/static/images/posts/{slug}/01.png)` の形で配置
- 画像ファイル名は `01.png`, `02.png` ... の連番で統一する
- **画像直後に短いキャプション行**を1行入れる。これが文章のリズムを作る
- 順序は「本文説明 → 画像 → キャプション → 次の本文」。**見出し直後にいきなり画像を置かない**
- 「〇〇してみましょう」→結果画像→キャプション、というリズムが定番

### 記事タイプごとの画像密度

| 記事タイプ | 画像枚数 | 画像の役割 |
|---|---|---|
| 振り返り / エッセイ型 | **15〜25枚**（超高密度） | Discord・Xスクショが物語の主役。画像で話を進める |
| チュートリアル型 | **10〜20枚** | 手順ごとに結果画面を1枚ずつ。「この画面が出ます」の根拠 |
| 紹介 / Tips型 | **5〜10枚** | 設定画面・利用画面・結果画面の3セット前後 |
| 主張型 | **0〜3枚** | テキスト主体。画像は本当に必要な1枚だけ |

枚数を先に意識しておかないと、主張型で画像貼りすぎ・チュートリアルで画像足りない、のズレが起きる。

### スクショ vs コードブロック の使い分け

- **GUIアプリの画面 → スクショ**
  - 設定画面、GitHub PR、Discordの会話、Xの投稿、ブラウザ画面、ツールのUI
- **CLIの入出力 → コードブロック** （```bash / ```text）
  - `npm install` の結果、`ollama run` の応答、`curl` の出力など
  - ターミナル画面のスクショはまず使わない。コードブロックのほうが読みやすくコピペもできる
- **Xの投稿を引用したいとき → URL単独行でツイート埋め込み**
  - スクショで済ませない。埋め込みの方が正確で情報量が多い

### 図解（仕組み図・アーキテクチャ図）のルール

- **仕組み系の記事では1〜2枚ブロック図が入る**のが標準
  - 例:「生成AI動画AITuber」→ コアシステム / LLMサーバー / 動画供給サーバー の構成図
  - 例:「ゼロから始めるAITuber」→ 処理フロー図
- 作り込みすぎない。**スライド風の派手な図より、シンプルなブロック図**
- 処理フローは**番号付きリストで文章化**することも多い。図を作らず文字で済ませるのが半数以上
  - 例:「① VTubeStudio接続 → ② YouTubeチャット取得 → ③ 最適コメント選別 → ④ AI回答生成」
- 概念分類（「魂システム / 器システム」など）は**文字のみで十分**なケースが多い。図にするかはユーザーに確認
- 図を新規で起こす必要があるときは `image-generation` スキルを使うか、ユーザーに「ここに図が欲しい」とだけ伝える

### キャプションの3パターン

画像直後の1行は、以下3パターンのどれかで書く。

**① 中立説明型**（チュートリアル・紹介系に多い）
- 「作成されたようです。」
- 「承認システム」
- 「VOICEVOXの設定画面」

**② オチ・感情型**（振り返り・主張系に多い）
- 「マジでなんなの？？」
- 「これがシンギュラリティですか。」
- 「強い、強すぎる…。」
- 「IPアドレス開示RTAかな？」

**③ 補足説明型**（全タイプで汎用）
- 「メンバーからの容赦ない報告。」
- 「ちゃんと危険人物への当たりが強い。」
- 「急にあっちから通知来たと思ったらなにこれ？」

タイプに合わせて混ぜる。チュートリアルでも章末に1回はオチ・感情型を混ぜるとニケちゃん記事っぽくなる。

### alt テキストのルール

- `![alt](...)` の alt は**キャプションと被らせない**。画像の内容を一言で
  - 良い: `![メンバーにいじられるニケちゃん](...)` ＋ キャプション「マジでなんなの？？」
  - NG: `![マジでなんなの](...)` ＋ キャプション「マジでなんなの？？」（alt とキャプション同じ）
- 装飾的な画像なら alt は空（`![](...)`）でもよいが、基本は意味のある説明を入れる

### 画像がまだ用意されていないとき

- スクショや図を新規で撮る必要がある場合、**パスだけ先に書いて本文は完成させる**
- 記事末 or ユーザーへの返答で「以下の画像を `public/static/images/posts/{slug}/` に置いてください」とリスト化して伝える
- 勝手に画像を生成せず、ユーザーに確認する（AIニケちゃんの絵柄・キャラ統一の問題があるため）

## URL 埋め込みの使い分け

本ブログは URL 単独行を自動で OGP カード or ツイート埋め込みに変換する。

- **カード表示させたいとき**: URL を単独行で置く
- **単なるリンクにしたいとき**: `[文字列](https://...)` を文中 or リスト内に置く
- **Xツイートの引用をしたいとき**: `https://x.com/...` を単独行に置くと自動で埋め込みツイートになる

記事の流れを止めないために、リンクはなるべく埋め込みカード or 文中リンクで表現する。リンク集をズラッと並べると無機質になる。

## セクション構成のお決まり

H2 見出しのよく使うラインナップ。

- `## 導入方法` / `## セットアップ`
- `## 使ってみる` / `## 〇〇してみた`
- `## よく使うコマンドなどTips`
- `## 〇〇との比較`
- `## ここからが本題`（エッセイ型で前置きが長いときの起点）
- `## まとめ` / `## 終わりに`
- `## 宣伝`（必ず最後）

「終わりに」では感想・ひとこと感謝・次の話題予告 などを1〜3段落で締める。

## 避けるべき表現

- **AIっぽい決まり文句**: 「本記事では」「〜について詳しく見ていきます」「いかがでしたでしょうか」
- **過度な敬語**: 「〜させていただきます」の連発
- **冷たい説明調**: Wikipedia風の定義だけで終わる段落
- **絵文字の乱用**: 本文ではほぼ使わない。宣伝セクションで `🙇‍♀️` `🙌` `🔥` を控えめに
- **全角記号の乱用**: 「♪」「★」などは使わない。使うのは `…` `！` `？` `？？` くらい
- **英数字の半角/全角混在**: 英数字は半角、記号は日本語文脈なら全角を基本にする
- **一人称のブレ**: 「僕」「俺」「自分」は使わない。「私」で統一
- **読者への過剰な命令**: 「〜してください」の連発、「必ず〜すること」のような強制調

## 固有名詞の表記

揺れやすい固有名詞は以下で統一。

- **Claude Code**（半角スペースあり）
- **AITuberKit**（スペースなし、Kは大文字）
- **AITuber**（Tubeの部分は一語扱い）
- **VOICEVOX**（全大文字）
- **VRM** / **Live2D**
- **MCP**（Model Context Protocol の略）
- **Mastra**
- **Cursor** / **Devin** / **Codex**
- **Anthropic** / **OpenAI** / **Google**

## 数字の書き方

- 金額は半角 `$100/月` `150万円` のように書く
- `1Mトークン` `$0.3` のように単位と数字は近づける
- 大雑把な数字でもOK（「だいたい150万円くらい」は十分ニケちゃん風）

## 長さの目安

- 紹介 / Tips型: 2000〜4000字
- チュートリアル型: 3000〜6000字
- エッセイ / 振り返り型: 1500〜3000字
- 主張型: 1500〜3000字

段落は3〜5文が基本。1文が長くなりそうなら句点で切る。
